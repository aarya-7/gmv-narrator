────────────────────────────────────────────────────────────────────────────────
  THE ONE-LINER  (say this first, every time)
────────────────────────────────────────────────────────────────────────────────

"I built an automated pipeline that detects when daily revenue moves abnormally,
figures out which part of the business caused it, and then uses an LLM to write
a plain-English summary — what happened, why, and what to do next. The key design
decision was keeping the LLM completely out of the math. It only writes. Every
number it uses was computed before it ever got involved."


────────────────────────────────────────────────────────────────────────────────
  WHAT IS GMV AND WHY DOES IT MATTER?
────────────────────────────────────────────────────────────────────────────────

GMV stands for Gross Merchandise Value — it's the total dollar value of products
sold on an e-commerce platform in a given period. It's the most important
top-line metric for any marketplace because it tells you how much buying activity
happened, regardless of what the platform itself keeps as revenue.

When GMV drops or spikes unexpectedly, someone needs to know about it fast —
was it a system failure? A promotion that worked too well? A bad actor placing
fake orders? This project automates the detection and diagnosis of those moments.


────────────────────────────────────────────────────────────────────────────────
  THE DATASET
────────────────────────────────────────────────────────────────────────────────

I used the Olist Brazilian E-Commerce dataset from Kaggle. It's real transaction
data from a Brazilian marketplace covering 2016 to 2018. I chose it specifically
because it has joinable segment dimensions — product categories and customer
regions — which means I can ask "which part of the business drove this move?"
Most public retail datasets don't have that.

I restricted the analysis window to January 2017 through August 2018 because the
early data was too sparse and the late data was cut off mid-trend. Using the full
range would have introduced noise and made results misleading.

I also injected three synthetic anomalies into a copy of the data — a big spike
in bed & bath products, a spike in sports products, and a regional drop in the
Southeast. I did this because real public datasets don't come with labeled
anomalies to test against. The detection pipeline never sees the injection code,
so it's genuinely blind to where the shocks are. I disclosed this clearly in the
project documentation.


────────────────────────────────────────────────────────────────────────────────
  THE ARCHITECTURE — THREE LAYERS, CLEAR BOUNDARIES
────────────────────────────────────────────────────────────────────────────────

This is the part interviewers care most about. The project has three completely
separate layers, and the rule is: each layer only does one thing.

  LAYER 1 — Statistical Detection
  ──────────────────────────────
  This layer looks at the daily total GMV series and asks: "Is today unusual?"
  It uses a method called rolling median + MAD (Median Absolute Deviation).

  Here's how it works in plain terms:
  - For each day, I look at the 28 days before it and compute the median GMV.
    That median is the "expected" value — the baseline.
  - I subtract the baseline from the actual value to get the residual
    (how far off today was).
  - I then compute how spread out those residuals have been over the past 28 days
    using MAD. MAD measures typical deviation without being thrown off by
    extreme values.
  - I divide the residual by the MAD and multiply by 0.6745 to get a robust
    z-score. If the absolute value of that z-score is above 3.5, I flag it as
    an anomaly.

  Why MAD instead of standard deviation?
  Because standard deviation is sensitive to outliers — the very thing you're
  trying to detect. If there's a huge spike on day one of your window, it
  inflates std-dev and makes the next anomaly look normal by comparison.
  MAD ignores extreme values and only cares about the median spread.
  That's a much more robust signal for this use case.

  Why detrend?
  The Olist data grows steeply over time. If you run the detector on raw GMV,
  it flags the entire growth curve as anomalous because today is always higher
  than last month. By detecting on residuals (actual minus rolling median),
  you're only measuring day-to-day surprises, not the trend itself.


  LAYER 2 — Segment Decomposition
  ────────────────────────────────
  Once a day is flagged as anomalous, the second layer asks: "Which segments
  drove this?" It uses a SQL query to compute each segment's dollar contribution
  to the total delta.

  Here's the logic:
  - For each segment (e.g. "sports & leisure"), I compute its 28-day average
    GMV. That's its baseline.
  - On the anomaly date, I compare its actual GMV to that baseline to get
    the segment's delta.
  - I then divide each segment's delta by the total delta to get its
    contribution percentage.

  So if total GMV dropped by R$15,000 and the Southeast region was responsible
  for R$11,000 of that drop, Southeast contributed 73% of the move.

  I wrote a test that validates this math before any LLM code runs. The test
  checks that all segment contributions sum to roughly 100% of the total delta.
  If they don't, the "why" layer is producing fiction — and the LLM would
  faithfully narrate that fiction. Catching it here is the whole point.


  LAYER 3 — LLM Narration
  ────────────────────────
  Only after detection and decomposition are complete does the LLM get involved.
  It receives a structured fact packet — a clean dictionary of pre-computed
  numbers and rankings. No raw data. No SQL results. No dataframes.

  The LLM (Claude Sonnet, via the Anthropic API) is told to write exactly three
  sections: What Happened, Why, and Recommended Action. The system prompt
  explicitly says: "Do not output any number that is not derivable from the
  input." That single constraint is what makes the output trustworthy.

  The LLM's job is language, not logic. It turns facts into sentences.
  Everything else was done before it got involved.


────────────────────────────────────────────────────────────────────────────────
  WHY THIS ARCHITECTURE? (the answer that impresses)
────────────────────────────────────────────────────────────────────────────────

"The fundamental problem with using an LLM to analyze data is hallucination.
If you give a language model raw numbers and ask 'what's unusual here?', it will
produce confident-sounding answers that may be mathematically wrong. You can't
audit them. You can't explain them. You can't put them in front of a CFO.

My architecture solves this by separating concerns completely. Detection is
statistical and auditable — you can reproduce the z-score calculation by hand.
Decomposition is pure SQL algebra — every number is traceable to a database row.
The LLM only receives facts that have already been verified, and it's explicitly
told not to invent any new numbers.

Every narrative in this project is fully auditable. I save the context packet
alongside the narrative output, so a reader can verify line by line that the
LLM didn't drift from the facts."


────────────────────────────────────────────────────────────────────────────────
  THE TECH STACK AND WHY
────────────────────────────────────────────────────────────────────────────────

Python 3.12
  - Industry standard for data engineering. Used for all pipeline logic.

PostgreSQL
  - The decomposition query is the analytical core of the project. Running it
    in SQL means the computation is set-based, auditable, and can be inspected
    directly in a database client. No black boxes.

Anthropic SDK (Claude Sonnet)
  - Claude Sonnet is strong at structured writing tasks and is cost-effective
    for a pipeline that runs occasionally, not in real time.

pandas + numpy
  - Used only in the detection layer for rolling window calculations.
    Not touched by the LLM layer.

pytest
  - The contribution-sum test is the gate that must pass before the LLM
    layer is allowed to run. This enforces the separation of concerns
    even at the code level.

No orchestration frameworks, no dbt, no Airflow. The build is intentionally
lean so every moving part is something I can explain end to end.


────────────────────────────────────────────────────────────────────────────────
  WHAT THE PIPELINE ACTUALLY PRODUCES
────────────────────────────────────────────────────────────────────────────────

Running the backfill across the full 2017-2018 window detected 12 anomaly days.

  - 11 spikes, 1 drop
  - 5 high severity, 7 medium severity
  - The Black Friday cluster (Nov 20–28, 2017) was the most dramatic:
    November 24th had a z-score of +29.79 with GMV 558% above baseline.
    Every region moved upward simultaneously, which is the signature of a
    platform-wide promotional event.
  - All three synthetic shocks were detected cleanly with z-scores of
    +4.28, +8.34, and -3.82 respectively.

Each detected anomaly produced:
  - A markdown file with the full context packet and narrative side by side
    (so anyone can audit the LLM's output against the raw facts)
  - An entry in a structured JSONL log used to look up historical context
    for future narratives ("a drop of similar severity occurred 214 days ago")
  - One page in a professional PDF report with metrics, driver tables,
    and the three narrative sections colour-coded by type


────────────────────────────────────────────────────────────────────────────────
  COMMON INTERVIEW QUESTIONS — ANSWERS READY TO GO
────────────────────────────────────────────────────────────────────────────────

Q: "Why not just use an LLM to detect the anomalies too?"

A: "Because LLMs are not calculators. They can hallucinate statistics, confuse
   correlation with causation, and produce confident-sounding wrong answers.
   Statistical anomaly detection with a defined threshold is reproducible —
   you can run the same numbers and get the same answer every time.
   An LLM detecting anomalies cannot make that guarantee."


Q: "Why MAD instead of standard deviation?"

A: "Standard deviation is sensitive to outliers — which is exactly the problem
   when you're trying to detect outliers. A single massive spike in the
   trailing window inflates std-dev and widens the detection band, potentially
   hiding the next anomaly. MAD is resistant to this because it's based on
   the median of absolute deviations, not the mean. It stays stable even when
   there are extreme values in the window."


Q: "How do you know the decomposition is correct?"

A: "I wrote a test that validates it before any LLM code runs. The test injects
   a known shock into a specific segment, runs the decomposition, and checks
   two things: (1) all segment contributions sum to roughly 100% of the total
   delta, and (2) the segment I injected the shock into appears in the top 3
   contributors. If either assertion fails, the pipeline stops. The gate has
   to pass before the narration layer is even reachable."


Q: "What would you do to productionise this?"

A: "A few things. First, I'd replace the rolling average baseline in the
   decomposition with the same rolling median I use in detection, so both
   layers are consistent. Second, I'd add segment-level detection — right now
   I only flag days where the total GMV is anomalous. You can have a category
   crash while the total looks flat because another category offset it.
   Third, I'd add a mix-vs-rate decomposition to split each segment's delta
   into 'their share of orders changed' versus 'their revenue per order
   changed.' That's the genuinely advanced version of the 'why.' And finally,
   I'd wire the outputs to a Slack webhook so the narrative lands in a channel
   automatically rather than requiring someone to open the PDF."


Q: "Why did you inject synthetic anomalies instead of using real ones?"

A: "Real public datasets don't come with labeled ground truth for anomalies.
   Without injecting known shocks, I have no way to write a test that says
   'the detector should have found this.' The synthetic shocks give me a
   reproducible test harness. I placed them on dates that were already
   near-anomalous in the clean data, so the required multipliers are
   realistic rather than absurd. And I disclosed them clearly — the
   detection logic is completely blind to the injection code."


Q: "Why PostgreSQL instead of pandas for the decomposition?"

A: "The decomposition is fundamentally a set operation — joining baselines to
   actuals, computing deltas, ranking by absolute value. SQL is the right tool
   for set operations. It's readable, auditable, and any analyst with database
   access can inspect the query and verify the math independently. If I did it
   in pandas, the logic is buried in Python code that requires running the
   notebook to verify. The SQL file is self-documenting."


Q: "What was the hardest part?"

A: "The hardest part was shock calibration. My first instinct was to inject
   35-40% drops into individual product categories. But even a 35% drop in one
   category that represents only 7% of total daily GMV only moves the total
   by about 2.5% — well below the z=3.5 detection threshold given the natural
   day-to-day volatility of the data. I had to analyse the rolling MAD for
   every date in the series, find dates where the surrounding volatility was
   low enough that a realistic shock would cross the threshold, and calibrate
   the multipliers accordingly. That gap between 'this sounds like a big shock'
   and 'this is actually detectable' is a real production problem too — it's
   exactly why you need to tune detection parameters against known events."

────────────────────────────────────────────────────────────────────────────────
  THE THING THAT MAKES THIS DIFFERENT FROM A TYPICAL LLM PROJECT
────────────────────────────────────────────────────────────────────────────────

Most LLM portfolio projects are wrappers. Someone plugs an API key into a chat
interface and calls it an "AI product." This project is different because the
LLM is not the system — it's the last mile of the system.

The value is in the statistical layer that nobody can hallucinate around,
the SQL layer that any analyst can audit, and the test that gates the whole
thing before any AI gets involved. The LLM writes. The math is airtight.
That's an architecture you could actually deploy in production.
